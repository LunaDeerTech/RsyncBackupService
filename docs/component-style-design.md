# Rsync Backup Service — 前端组件样式设计

---

## 1. 概述

本文定义 Rsync Backup Service 自定义前端组件样式库的视觉方向、样式 token 体系、组件分工、状态语义与验证规则，用于指导后续组件实现与页面落地。

该样式库服务于以下产品场景：

- 备份实例列表与详情
- 策略与存储目标管理
- 运行中任务、实时进度与统计卡片
- 恢复确认、危险操作确认与通知
- 审计日志、历史记录与高密度表格

设计目标不是构建展示型科技界面，而是在备份管理后台中建立稳定、可信、可长期使用的品牌识别。

## 2. 设计目标与非目标

### 2.1 设计目标

- 建立鲜明但不过度张扬的品牌风格，区别于通用后台模板
- 在浅色与深色主题下保持一致的材料逻辑与状态语义
- 优先保障表格、表单、日志与状态面板的可读性和可操作性
- 让运行中、成功、警告、失败、禁用、待确认等状态可以被快速扫视识别
- 通过轻流动动效、分层表面和 `Cyan Mint` 主品牌色建立统一气质

### 2.2 非目标

- 不追求大面积强发光、重霓虹或持续炫技动效
- 不将高密度页面卡片化到影响信息效率
- 不使用品牌主色去承担错误或危险语义
- 不为视觉个性牺牲键盘焦点、对比度和状态可辨识性

## 3. 选定方向

### 3.1 风格结论

本项目的组件样式方向确定为 `Balanced Flux`。

它基于 `Flux Layers` 的分层与流动感，但将表达强度控制在工具产品可长期使用的范围内。整体风格采用中等信息密度、中小圆角、轻流动动效和中等玻璃感，并以 `Cyan Mint` 作为主品牌色方向。

### 3.2 关键词

- 高可信工具界面
- 流动分层，而非炫目科技风
- 冷静、清洁、偏系统型的品牌气质
- 深浅主题并重
- 状态优先、品牌辅助

### 3.3 核心原则

- 品牌感通过面层节奏、主色、轻渐变和局部透明材质体现，不通过大面积装饰体现
- 高频操作组件保持更稳的实体感，高层级状态组件允许更柔和的表面处理
- 动效只服务于层级理解、焦点引导和状态变化，不做表演性运动

## 4. 视觉系统架构

### 4.1 Token 分层

组件样式库采用四层 token 架构：

1. `Base tokens`：定义原始色值、圆角、描边、阴影、透明度、间距和时长
2. `Semantic tokens`：定义 `surface`、`text`、`border`、`primary`、`success`、`warning`、`error`、`info`、`focus-ring` 等语义值
3. `Component tokens`：为按钮、输入框、表格、对话框、标签、进度条等组件绑定具体样式变量
4. `State tokens`：定义 hover、active、focus、selected、disabled、loading、danger 等状态变化

所有样式实现都应优先引用 token，而不是在组件中直接写死颜色、阴影或圆角值。

### 4.2 主题模型

浅色主题与深色主题不能简单反相，而应保持相同的材料逻辑：

- 深色主题：冷中性背景 + 3 到 4 级 surface 分层 + 少量青色高光
- 浅色主题：冷白背景 + 浅灰蓝 surface 分层 + 边框与弱阴影构建层次

无论主题如何切换，都应保持相同的组件层级关系、相近的视觉重量和一致的交互反馈。

### 4.3 几何与材料规则

- 主体圆角采用中小圆角体系
- 默认圆角建议：按钮 `8px`，输入与选择器 `10px`，卡片 `12px`，对话框 `14px`
- 描边优先于厚阴影，阴影只用于抬升层级，不用于定义轮廓
- 毛玻璃只出现在浮层、运行态摘要卡和图表概览，不进入高密度表格和表单主体

## 5. 色彩体系

### 5.1 品牌与语义色

以下为首版候选色板基线：

| Token | 值 | 用途 |
|------|------|------|
| `primary-300` | `#9BEAFF` | 浅色高亮、选中背景、轻强调 |
| `primary-500` | `#63D9FF` | 主品牌色、主按钮、主交互高亮 |
| `primary-600` | `#2FC7F0` | 主按钮 hover、激活描边 |
| `accent-mint-400` | `#7EF2D4` | 运行态强化、进度渐变辅助色 |
| `success-500` | `#5DCC96` | 成功状态 |
| `warning-500` | `#F5BE58` | 警告状态 |
| `error-500` | `#E77483` | 错误与危险状态 |
| `info-500` | `#70B7FF` | 信息状态 |

规则如下：

- `primary` 仅承担品牌与主交互语义
- `success`、`warning`、`error`、`info` 必须各自独立，不与品牌主色混用
- 危险操作使用 `error` 体系，不允许借用青色品牌语言进行美化

### 5.2 深色主题基线

| Token | 值 | 用途 |
|------|------|------|
| `bg-canvas` | `#0E1726` | 页面主背景 |
| `bg-subtle` | `#111D2D` | 次级背景、侧栏 |
| `surface-1` | `#142133` | 基础卡片/输入背景 |
| `surface-2` | `#18283D` | 抬升一层的卡片/面板 |
| `surface-3` | `#1E3147` | 浮层/高优先层 |
| `border-default` | `#294059` | 默认描边 |
| `text-strong` | `#EAF5FF` | 主文字 |
| `text-muted` | `#96ABC1` | 次级文字 |

### 5.3 浅色主题基线

| Token | 值 | 用途 |
|------|------|------|
| `bg-canvas` | `#F4F8FC` | 页面主背景 |
| `bg-subtle` | `#EDF3F8` | 次级背景 |
| `surface-1` | `#FFFFFF` | 基础卡片/输入背景 |
| `surface-2` | `#F6FAFD` | 抬升一层的卡片/面板 |
| `surface-3` | `#EAF2F8` | 选中背景/弱强调背景 |
| `border-default` | `#D6E3EC` | 默认描边 |
| `text-strong` | `#102033` | 主文字 |
| `text-muted` | `#62758A` | 次级文字 |

### 5.4 状态与对比规则

- 所有关键状态都必须通过颜色之外的方式再次表达，如图标、标签文字或形状变化
- 在浅色与深色主题下，主文字、禁用态、错误提示、焦点环都应遵循一致的对比度逻辑
- 运行中态可以使用 `primary + accent-mint` 渐变，但失败态和危险态不能使用渐变弱化风险感

## 6. 排版、图标与数据展示

### 6.1 排版原则

- 正文应使用支持中文界面的现代无衬线字体
- 标题、卡片数值与关键状态数据可稍微提高字重，但不依赖极端粗细对比制造层级
- 英文缩写、速率、路径片段、时间和任务编号可使用等宽数字或等宽字体变体，提升扫描效率

### 6.2 字号建议

- 页面标题：`28px`
- 区块标题：`20px`
- 卡片标题：`16px`
- 正文：`14px`
- 辅助文字：`12px`
- 指标大数值：`24px` 到 `32px`

### 6.3 图标原则

- 图标风格应简洁、线性、轮廓明确
- 状态图标优先承担识别任务，不做复杂拟物
- 运行中、成功、警告、失败图标在所有页面必须保持一致含义

## 7. 组件家族设计规则

### 7.1 操作型实体组件

适用范围：按钮、输入框、密码输入、选择器、开关、标签输入、分页器、表单容器、表格单元。

规则：

- 保持实体感，不使用明显玻璃效果
- 以清晰描边、焦点环和短促状态变化作为主要反馈
- hover 主要通过亮度和边框变化表达，避免大范围位移
- disabled 态降低对比度并减少阴影，但仍保持可辨识轮廓

### 7.2 状态与概览组件

适用范围：统计卡片、运行态卡片、进度条、通知、时间线、抽屉、对话框、详情侧板。

规则：

- 允许轻渐变、局部透明和更明显的层级抬升
- 允许在运行中态中使用 `primary + accent-mint` 的轻流动色带
- 不允许通过持续发光替代文字说明
- 高风险确认弹窗应收敛装饰，优先突出警告与后果说明

### 7.3 表格与日志组件

适用范围：实例列表、备份历史、恢复历史、审计日志。

规则：

- 作为视觉纪律中心，必须保持结构稳定和高扫描效率
- 行 hover、选中与筛选态应克制，优先使用弱背景变化与边界加强
- 不在整张表格中使用大面积渐变或毛玻璃
- 状态列统一使用标签、图标和简洁文本组合表达

### 7.4 关键组件行为建议

- `Button`：主按钮采用品牌色，次按钮回归中性面层，危险按钮使用独立 error 体系
- `Input / Select / Textarea`：统一焦点样式，优先边框变化 + 外部 focus ring
- `Card`：默认实体卡，统计卡与运行态卡允许更柔和的材质处理
- `Progress`：支持运行态渐变，但要同时显示百分比、速率或剩余时间之一
- `Tag / Badge`：作为全局语义组件，确保状态词视觉统一
- `Modal / Dialog`：可使用轻透明头部或背景模糊，但正文区保持高对比与稳定排版

## 8. 动效系统

### 8.1 动效策略

动效风格采用“轻流动型”：

- 微交互短促直接
- 层级切换柔和但不拖沓
- 运行中态有连续感，但不形成炫目噪声

### 8.2 时长建议

- Hover / focus：`120ms` 到 `160ms`
- 弹层进入/退出：`180ms` 到 `220ms`
- 页面局部切换：`200ms` 到 `260ms`
- 进度与骨架的循环动画：`900ms` 到 `1400ms`

### 8.3 允许与禁止

允许：

- opacity + translate 的短距离过渡
- 进度条内部的缓慢流动
- 卡片与弹层的轻微抬升

禁止：

- 长距离飞入飞出
- 高频闪烁
- 无限强发光边框
- 在表格与表单主体中使用连续运动背景

## 9. 页面应用指导

### 9.1 仪表盘

- 允许使用最完整的 `Balanced Flux` 语言
- 统计卡、运行中任务和时间线可以承担更多品牌表达
- 图表和进度条是主要视觉焦点

### 9.2 列表与配置页面

- 强调清晰结构和高密度阅读
- 品牌表达集中在筛选控件焦点态、主操作按钮和局部状态标签
- 主体区域以稳定 surface 和清楚边界为主

### 9.3 恢复与危险操作页面

- 优先保证风险信息、影响范围和确认动作最突出
- 弱化品牌装饰，强化 warning / error 语义
- 二次认证输入框和确认按钮应具有最明确的焦点与层级

### 9.4 审计日志页面

- 以表格与时间线混合方式呈现时，仍以信息扫描效率为最高优先级
- 状态颜色、动作标签和时间信息应保持一致秩序
- 不引入与任务运行态相同强度的动态视觉

## 10. 可访问性与验证

### 10.1 可访问性要求

- 任何关键状态不能只依赖颜色表达
- 键盘焦点必须清楚可见，不能因为视觉收敛而被削弱
- 表单错误提示必须同时包含颜色、图标和文本提示
- 浅色与深色主题都要验证正文、辅助文字、描边和状态提示的对比可读性

### 10.2 验证范围

样式设计落地后至少执行以下验证：

1. Token 一致性检查：验证浅深主题和语义色未出现交叉污染
2. 组件视觉回归：覆盖按钮、输入、表格、卡片、标签、进度条、对话框、通知等核心组件
3. 页面级场景评审：覆盖仪表盘、实例详情、备份历史、恢复确认、审计日志、运行中任务列表

### 10.3 验证标准

- 页面在深浅主题下都保持相同层级逻辑
- 高密度页面没有因品牌表达导致阅读效率下降
- 危险态、错误态和警告态与品牌主色不存在混淆
- 动效不会影响输入、筛选、表格浏览和风险确认效率

## 11. 实现边界

- 首版实现优先覆盖核心组件清单：Button、Input、Textarea、Select、Switch、Table、Modal、Dialog、Form、Card、Tag、Badge、Progress、Toast、Notification、Tabs、Breadcrumb、Sidebar、Layout、Icon、Empty、Spinner、PasswordInput、Timeline
- 样式系统优先以 CSS 变量或等价 token 方案驱动
- 首版不引入多品牌皮肤，仅实现一套 `Balanced Flux` 语言下的浅深主题
- 首版不追求复杂插画、3D 纹理或装饰性图形系统

## 12. 结论

Rsync Backup Service 的自定义组件样式库应采用 `Balanced Flux` 方案：以 `Cyan Mint` 为主品牌色，以中小圆角和中等密度建立稳定骨架，以轻流动和局部分层材料建立品牌识别，以严格独立的语义色和克制的状态动效保证工具产品的可靠性。

该方案的判断标准很明确：它必须既能让仪表盘和运行态任务拥有鲜明气质，也能让实例列表、备份历史、恢复确认和审计日志这些高频、高风险页面保持冷静、清晰和高效。
